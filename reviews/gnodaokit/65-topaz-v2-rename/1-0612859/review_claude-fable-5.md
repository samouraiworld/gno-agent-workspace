## Summary

The port commit (15dbc83) is [#64](https://github.com/samouraiworld/gnodaokit/pull/64)'s first commit unchanged: `cur realm` threading down the Execute path, crossing-wrapper removal, gno pin bump to 2c7f1abe. The rename commit (0612859) is mechanical: 3 gnomod module lines to `…/v2`, 44 internal import repoints, 8 avl repoints to the vendored [`p/samcrew/avl`](https://github.com/samouraiworld/samcrew-deployer/blob/9b2b22e/deps/avl/gnomod.toml#L1) (two-value `Get` frozen), and one forced alias in [`gno/p/basedao/utils.gno:9`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/basedao/utils.gno#L9) keeping the canvas style tree typed `*p/nt/avl/v0.Tree` because genesis [`p/demo/svg` declares `Style` with that type](https://github.com/gnolang/gno/blob/fc4052651/examples/gno.land/p/demo/svg/svg.gno#L15).

The rename premise fails at the launch tip. gnolang/gno [PR 5945](https://github.com/gnolang/gno/pull/5945) builds topaz-1 as a fresh chain from a curated package set, generated by [`gen-genesis.sh`](https://github.com/gnolang/gno/blob/fc4052651/misc/deployments/topaz.gno.land/gen-genesis.sh#L643) over the repo's own [`examples/` tree](https://github.com/gnolang/gno/blob/fc4052651/misc/deployments/topaz.gno.land/gen-genesis.sh#L421). At the announced launch tip fc4052651, `p/samcrew/{daokit,basedao,daocond}` sit in [`examples/quarantined/`](https://github.com/gnolang/gno/tree/fc40526511474e40b8a66419f5ba28255085bc08/examples/quarantined/gno.land/p/samcrew), outside every deployable path. Reproducing the set from the script's own [`FILTERED_PACKAGES`](https://github.com/gnolang/gno/blob/fc4052651/misc/deployments/topaz.gno.land/gen-genesis.sh#L66) yields exactly 90 packages whose only samcrew entries are piechart, tablesort and urlfilter. The collision existed only under the abandoned test13-hardfork design; the original module paths are free on topaz-1.

The avl repoint and the `ntavl` alias survive the premise change on their own: genesis `p/nt/avl/v0` has [single-value `Get`](https://github.com/gnolang/gno/blob/fc4052651/examples/gno.land/p/nt/avl/v0/tree.gno#L58) while gnodaokit uses two-value `Get` at 12+ call sites, and the svg boundary must keep the genesis tree type. `realmid` is correctly left unrenamed: it is absent from the reproduced set.

Timing caveat, not a reason to keep the renames: topaz-1 is not yet publicly inspectable and the deployer carries its own [merge gate](https://github.com/samouraiworld/samcrew-deployer/blob/9b2b22e/CHANGELOG.md?plain=1#L15) holding until the launched chain's `build_version` confirms fc4052651. The strip lands on the same gate.

## Warnings (should fix)
- **[caller picks the realm the DAO acts as]** [`gno/p/daokit/daokit.gno:17`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/daokit/daokit.gno#L17) — `Execute(id, rlm)` takes the realm from its caller, so a DAO action crosses under whichever realm invoked it, not the DAO's own.
  <details><summary>details</summary>

  `Core.Execute` forwards `rlm` to the action handler ([`daokit.gno:91`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/daokit/daokit.gno#L91)), and `NewEditProfileHandler` writes with `setter(cross(rlm), k, v)` ([`basedao/actions.gno:146`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/basedao/actions.gno#L146)). `Execute` is on the exported `daokit.DAO` interface, so any holder supplies its own `cur` and the write lands on that realm's profile. Ran it: a foreign realm's EditProfile execution recorded `gno.land/r/test/caller|Bio=written-by-caller`, while the DAO's own `basedao.New` init on the same DAO recorded `gno.land/r/test/daorlm`; see [tests](tests/README.md). The same realm flows into `InstantExecute` ([`daokit.gno:34-37`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/daokit/daokit.gno#L34-L37)), which `NewInstantExecuteAction` aims at a caller-supplied target DAO ([`actions.gno:136`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/daokit/actions.gno#L136), field commented "Target DAO to execute on"), so a parent DAO executing on a sub-DAO makes the sub-DAO's profile writes land on the parent. The deleted [`crossing.gno`](https://github.com/samouraiworld/gnodaokit/blob/b833296/gno/p/daokit/crossing.gno#L36-L40) routed these through the DAO realm's own `crossFn`, so the realm was the DAO's rather than the caller's; nothing replaced it, and `DAOPrivate.Realm` is a plain `runtime.Realm` tuple, not a usable realm capability. Latent in the shipped demos only because they keep `localDAO` unexported. Fix: have the handler cross under the DAO's own realm instead of the one the caller passed.
  </details>
- **[cross-realm callers have no identity]** [`gno/p/basedao/basedao.gno:69`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/basedao/basedao.gno#L69) — `CallerIDFn` takes no realm, so a caller from another realm resolves to the empty string and `MembersStore.AddMember` will happily store that as a member address.
  <details><summary>details</summary>

  `Execute` gained realm threading but `Propose` and `Vote` did not, and `CallerID` stays a bare `func() string` defaulting to `realmid.Previous` ([`basedao.gno:120`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/basedao/basedao.gno#L120)), which reads [`unsafe.PreviousRealm()`](https://github.com/gnolang/gno/blob/fc4052651/gnovm/stdlibs/chain/runtime/unsafe/unsafe.gno#L26). That call is documented to identify the outermost-crossing realm rather than the immediate caller in a non-crossing helper, and the `daokit.DAO` methods are non-crossing. Ran it: a DAO queried through its interface from a different realm reports `CallerID()` as the empty string; see [tests](tests/README.md). `assertCallerIsMember` ([`basedao.gno:244`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/basedao/basedao.gno#L244)) then tests `IsMember("")`, and [`AddMember`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/basedao/members.gno#L184) validates neither emptiness nor address shape. With one empty-address member present, a foreign realm proposed, voted and executed on the DAO in the same run. Without it the gate panics `caller is not a member`, which is fail-closed but also means a DAO cannot identify a calling realm at all, so the multi-realm support the deleted `crossing.gno` provided is gone. Fix: reject empty and malformed member addresses in `AddMember`, and give `CallerIDFn` a realm to read so a calling realm is identifiable.
  </details>

## Nits
- **[lambda actions cannot cross]** [`gno/p/daokit/actions.gno:93-94`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/daokit/actions.gno#L93-L94) — `NewExecuteLambdaHandler` discards its realm into `_` and invokes a `func()` payload, so a lambda action can never do what the threading was added for, while `NewInstantExecuteHandler` two functions below forwards it. Nothing in the type or the doc comment says lambdas are the non-crossing kind.
- **[New ignores the realm it was given]** [`gno/p/basedao/basedao.gno:128`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/p/basedao/basedao.gno#L128) — `New` stores `Realm: unsafe.CurrentRealm()` although it holds an authoritative `rlm realm` parameter that cannot lie. The two agree in the demos, where `New` is only called from `init(cur realm)`, so this is latent. `DAOPrivate.Realm` feeds the private-extension gate, so a borrowed call site would diverge.
- **[rename sweep missed a file]** [`gno/r/daodemo/README.md:13-15`](https://github.com/samouraiworld/gnodaokit/blob/0612859/gno/r/daodemo/README.md?plain=1#L13-L15) — the sources list keeps unsuffixed `/p/samcrew/{daokit,basedao,daocond}` links while the code renames to `/v2`.
