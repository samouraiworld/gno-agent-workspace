## What changed since round 1

The PR was substantially reworked between `44a8373` and `75153e206`, mostly in response to @thehowl's review:

- **Simulation behavior reverted to server-neutral.** Round 1 reviewed a version where `baseapp`/`ante` installed the consensus max-gas meter during simulation. Per [@thehowl's design note](https://github.com/gnolang/gno/pull/4951#issuecomment-) the author reverted that (commit `949280237`) and moved the gas-cap rewrite entirely to the client ([`buildSimulationTxBytes`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L179) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L179)). `baseapp`/`ante` now only change the OOG *message*, not the meter.
- **OOG message centralized.** Round 1's three ad-hoc OOG strings collapsed into [`store.OutOfGasLog`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/store/types/gas.go#L132) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/store/types/gas.go#L132) (@thehowl's "put it in a shared place" comment). Exported via [`store/exports.go:40`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/store/exports.go#L40) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/store/exports.go#L40).
- **`margin++` round-up.** `suggestedGasWanted` now rounds the 5% margin up via `if gasUsed%20 != 0 { margin++ }` ([`broadcast.go:200-205`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L200-L205) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L200-L205)), addressing @thehowl's suggestion.
- **Suggestion folded into the estimate line.** Round 1 emitted `suggested gas-wanted ...` as a separate clause; per [@thehowl's "feels unnatural" comment](https://github.com/gnolang/gno/pull/4951#discussion_) it now reads `estimated gas usage: N (suggested, with 5% margin: M)` inside `estimateGasFee` ([`broadcast.go:234,241`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L234-L241) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L234-L241)).
- **Fetch-failure now warns instead of silently swallowing.** [`resolveMaxGas`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/maketx.go#L371) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/maketx.go#L371) prints a `warning:` and falls back to maxGas=0 (keep user's tx bytes), addressing @thehowl's maketx.go:191 thread.
- **`context.Background()` with documented 60s timeout** ([`broadcast.go:125-127`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L125-L127) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L125-L127)), per @thehowl's maketx.go:293 note.
- **maxGas==0 fallback concern resolved by construction.** @thehowl worried the `MaxInt64` fallback would trip the ante's gas-wanted cap on fetch failure. It can't: fetch failure returns `0`, and [`buildSimulationTxBytes`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L180-L185) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L180-L185) returns the original bytes unchanged for `case 0`. `MaxInt64` is only reached on `maxGas == -1` (chain explicitly has no limit).
- **New tests.** `maketx_test.go` (186 lines) covers `buildSimulationTxBytes`, `resolveMaxGas`, `appendSuggestedGasWanted`, `handleDeliverResult`, and all five `OutOfGasLog` branches; `baseapp_test.go` adds `TestOOGLogBeyondGasWanted` and `TestOOGLogUsesMaxBlockGas`.

## Warnings (should fix)

- **[suggestion derived from partial gas on a failed simulation]** [`tm2/pkg/crypto/keys/client/broadcast.go:157`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L157) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L157) — `appendSuggestedGasWanted(res)` runs on the error path, so a simulation that OOG'd emits `suggested gas-wanted` computed from gas-until-panic.
  <details><summary>details</summary>

  When the simulation errors (server OOG on the rewritten tx, or — when the consensus fetch failed and `simulateMaxGas==0` — the user's own under-set `GasWanted`), `hasError` is true, the `cfg.DryRun && !hasError` branch is skipped, and execution falls to [`appendSuggestedGasWanted(res)`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L157) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L157) before the `if hasError { return }`. `res.DeliverTx.GasUsed` is then the partial gas consumed before the panic, and the suggestion is `partial + 5%` — a number the user cannot act on. This is the round-1 Critical, unchanged. The author argues ([broadcast.go:157 thread](https://github.com/gnolang/gno/pull/4951#discussion_)) that the error path is "usually the most useful case" to show a suggestion. That holds only when the rewrite succeeded and the server completed; on a genuine OOG the number is gas-until-panic, not gas-needed. Verified with an adversarial test ([`tests/appendsuggested_on_oog_test.go`](tests/appendsuggested_on_oog_test.go)): `GasUsed=47, Error set` yields `Info = "suggested gas-wanted (gas used + 5%): 50"`. Fix: gate the call on `res.DeliverTx.Error == nil`, or annotate the suggestion as unreliable when the run errored.
  </details>

When a broadcast tx OOGs in DeliverTx with `gasWanted < maxGas` (the common case: gasWanted in the millions, block maxGas in the hundreds of millions), `OutOfGasLog` takes the hint branch ([`gas.go:139-143`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/store/types/gas.go#L139-L143) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/store/types/gas.go#L139-L143)) and tells an operator who already broadcast to go simulate. `result.Log` for a DeliverTx lands in block events, indexers, and explorers. `TestOOGLogBeyondGasWanted` ([`baseapp_test.go:1066`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/sdk/baseapp_test.go#L1066) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/sdk/baseapp_test.go#L1066)) asserts this hint on a DeliverTx, locking the behavior in. The author addressed the *client* side (the synthesized simulation OOG passes `false` at [`broadcast.go:146`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L146) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L146)) but not the server side. - **[Addp panic on near-MaxInt64 gasUsed]** [`tm2/pkg/crypto/keys/client/broadcast.go:205`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L205) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L205) — `suggestedGasWanted` ends with `overflow.Addp(gasUsed, margin)`, which panics on overflow; for `gasUsed > ~8.78e18` the CLI crashes.
  <details><summary>details</summary>

  `overflow.Addp` panics by contract ([`overflow.go:50-56`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/overflow/overflow.go#L50-L56) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/overflow/overflow.go#L50-L56)). @thehowl's `margin++` suggestion fixed the *margin* computation but the final `gasUsed + margin` add is still exposed. Reachable only via the `Block.MaxGas == -1` fallback path (`simulationMaxGasFallback = MaxInt64`, [`broadcast.go:38`](https://github.com/gnolang/gno/blob/75153e206/tm2/pkg/crypto/keys/client/broadcast.go#L38) · [↗](../../../../../.worktrees/gno-review-4951/tm2/pkg/crypto/keys/client/broadcast.go#L38)) plus a runaway tx that simulates to near-MaxInt64 gas — narrow, but `appendSuggestedGasWanted` runs on every simulation result including the error path.   </details>
