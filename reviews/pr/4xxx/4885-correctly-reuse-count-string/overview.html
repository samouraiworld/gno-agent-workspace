How the GnoVM counts string memory

PR #4885 · fix(gnovm): correctly reuse/count string in alloc and gc · explainer generated by Claude Opus 4.8
Plain-language overview · no review verdict here

The GnoVM meters how much memory a transaction uses, so it can stop one transaction from exhausting a node. Strings were counted wrong: shared text was billed several times. This PR makes every chunk of string bytes count exactly once, even when many strings point at it.

The problem in one picture

A Gno string is two parts: a small header (a pointer + length) and the backing (the actual bytes). Slicing or aliasing makes a new header but shares the same bytes, nothing is copied.

backing bytes (the real memory) where a slice header points
s := "ABCDEFGH" → one backing. s2 := s[3:] → a new header pointing at byte 3 of the same backing. The bytes exist once; the headers are just views.

Before (on master)

Every header was billed for the full backing on every GC visit. s and s2 each charge 8 bytes → 16. Aliasing looked like more memory than really existed. Safe but wrong: programs hit the limit too early.

After (this PR)

The backing is billed once, no matter how many headers view it. s + s2 charge the 8 backing bytes once, plus a tiny header each. The meter now matches reality.

The idea: count by containment, not by pointer

When a string is allocated, the VM records its backing as an address span [start, end). At garbage-collection time, for each string it asks: which span does this pointer fall inside? The first string to land in a span this cycle pays the full backing; everyone else in the same span pays nothing.

This is what makes a slice safe: s[3:] points at start+3, which is still inside the source's span, so it resolves to the same backing, even if the original s has already died. An earlier draft of this PR keyed on the exact pointer instead, and start+3 was never a key, so a surviving slice's bytes silently vanished from the budget. Containment removes that whole class of bug.

Why a slice charges the full backing

Go cannot free part of an array. While any slice points into a backing, the entire backing stays in memory. So holding s[3:] of a 1 MB string pins all 1 MB, and the meter charges all 1 MB, not just the slice's length.

Count-once-per-cycle

A GC cycle is one run of the collector. It is not periodic: it fires when an allocation would cross the per-transaction cap (500 MB), or when Gno code calls runtime.GC(). Each run wipes the running total and re-adds every live backing once. A per-span "last seen cycle" stamp is how the VM knows it already counted a backing this run; next run it counts fresh again. Spans nobody touched this run are dropped.

Concepts

TermPlain meaning
backingThe actual array of string bytes. Shared by every header that views it. Freed by Go only when nothing references it.
headerThe tiny {pointer, length} pair. Slicing makes a new one; the bytes are not copied.
span / rangeThe address extent [start, end) of one backing, recorded so the GC can attribute any interior pointer to it.
containment lookup"Which span is this pointer inside?" rather than "is this pointer an exact key?" Lets a slice resolve to its source.
GC cycleOne collector run. Triggered by hitting the memory cap or an explicit runtime.GC(), not by a timer.
gasCost unit, calibrated so 1 gas ≈ 1 nanosecond. A GC run is charged for the values it walks; allocation is charged for bytes reserved.
