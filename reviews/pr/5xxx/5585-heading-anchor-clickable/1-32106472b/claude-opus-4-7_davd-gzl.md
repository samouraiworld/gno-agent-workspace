## Summary

Closes #5579: rendered headings get auto-generated `id`s but the heading text itself is not clickable, so there is no quick way to copy a section permalink. This PR adds a goldmark extension (`ExtHeading`) that wraps each contiguous run of non-link inline children of every heading in `<a class="heading-anchor" href="#id">…</a>`. Inline links inside the heading are left untouched and rendered by their existing renderers, so nested `<a>` is impossible by construction.

Implementation:

- New AST transformer + inline node + node renderer in [gno.land/pkg/gnoweb/markdown/ext_heading.go](.worktrees/gno-fix-5579/gno.land/pkg/gnoweb/markdown/ext_heading.go).
- Transformer registered at priority 999 so it runs after [linkTransformer](.worktrees/gno-fix-5579/gno.land/pkg/gnoweb/markdown/ext_links.go#L91) (priority 500), which rewrites `Link` → `GnoLink` — `isLinkLike` relies on the final kind.
- Renderer emits `<a class="heading-anchor" href="#…">…</a>`; default goldmark heading renderer continues to emit `<h2 id="…">…</h2>`.
- CSS in [05-composition.css:654-671](.worktrees/gno-fix-5579/gno.land/pkg/gnoweb/frontend/css/05-composition.css#L654-L671): `color: inherit`, `display: inline` reset, `:focus-visible` outline. Built `main.css` regenerated.
- ADR at [gno.land/adr/pr5585_heading_anchor_links.md](.worktrees/gno-fix-5579/gno.land/adr/pr5585_heading_anchor_links.md) documents alternatives considered (always-wrap, sibling-only, sibling-when-link, goldmark-anchor, JS-only, aria-label).
- Demo realm [examples/gno.land/r/docs/markdown/markdown.gno:39-45](.worktrees/gno-fix-5579/examples/gno.land/r/docs/markdown/markdown.gno#L39-L45) showcases a heading with an inline link.

The PR went through three design rounds: (1) wrap the whole heading in a single `<a>` (broken on inline links), (2) sibling-mode empty anchor with a hover-`§` glyph (less discoverable), (3) per-non-link-run wrap (current). Commit history shows the churn.

Two mitigations:

  - At least add a `// CONTRIBUTORS:` comment near `isLinkLike` listing where to update when a new extension lands.

If anything ever runs the transformer twice on the same document, the second pass walks into the heading, sees the existing `headingAnchorNode` (not `isLinkLike`), and wraps it inside a *new* `headingAnchorNode` — nested anchor. Defensive: short-circuit when `c.Kind() == KindHeadingAnchor`, or assert single-pass via a `parser.Context` flag.

## Suggestions

- The transformer at priority 999 depends on linkTransformer at priority 500 having already run. That contract is verbal (in a comment). If someone bumps linkTransformer's priority above 999, `isLinkLike` would silently miss `Link` nodes that haven't been rewritten to `GnoLink` yet. Worth adding a parser-context check: assert in `Transform` that `linkTransformer` has run, or fold the priority dependency into a constant pair (`PriorityLinkTransformer = 500`, `PriorityHeadingAnchor = PriorityLinkTransformer + 499`).
- `display: inline` reset on `.heading-anchor` ([05-composition.css:662](.worktrees/gno-fix-5579/gno.land/pkg/gnoweb/frontend/css/05-composition.css#L662)) is needed because `.c-realm-view a` defaults to `inline-block`, which collapses trailing whitespace inside a wrapped run ("Title with " before an inline link). The relationship is non-obvious — the comment explains it, good. Consider also adding a unit-style CSS regression test (visual smoke) for that specific case.
- Squash the WIP/refactor commits into one or two clean commits before merge. Current sequence (8 commits, three of them refactors) makes the diff harder to follow than the final code.
- Image-in-heading click hijack: confirmed by `tests/adversarial_test.go:TestImageInHeadingClickHijack` — clicking an image inside a heading now sets `window.location.hash` instead of doing nothing. Document this in the ADR's Consequences section so the next maintainer doesn't think it's a bug.
- The ADR mentions rejecting `aria-label="Link to this section"` on the heading-anchor because it overrides the heading's accessible name. Worth also stating *what the current accessible name is* (the wrapped text, which equals heading text for plain headings, and a sub-string for headings with inline links). This affects screen-reader heading navigation — H key in NVDA/JAWS will announce the (possibly partial) text.

## Questions for Author

- Why priority 999 specifically? linkTransformer is at 500 — anything between 501 and the rendering pipeline would work. Pick the smallest gap that still leaves room for future transformers, or document why 999.
- Should `ExtHeading` register itself only when the goldmark instance has `parser.WithAutoHeadingID()` enabled? Currently it silently no-ops. A loud warning at extension registration time would catch misconfigured callers earlier.
- Consider exporting `KindHeadingAnchor` only — the type `headingAnchorNode` is unexported. External tooling (linters, doc generators) can't construct one. Intentional?
- Should there be a way to opt out of heading anchors per-document (e.g. via parser context flag) for cases where headings are rendered into a non-page context (Help view, RPC payload, …)? The ExtHeading is currently always-on inside `GnoExtension`.
