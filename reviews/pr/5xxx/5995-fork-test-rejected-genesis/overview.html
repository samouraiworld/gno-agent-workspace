PR 5995: who listens when the chain says no

Explainer for gnolang/gno#5995. Generated by claude-opus-4-8.

A chain upgrade replaces one running chain with a new one that starts from a hand-built genesis file. That file is large, generated by tooling, and only gets exercised for real once. So there is a tool, gnogenesis fork test, whose whole job is to boot a throwaway node against the file and tell an operator whether the real chain would start.

It said PASS on files the real chain refuses.

The chain of custody for a rejection

When a node starts for the first time, consensus asks the application to load the genesis. The application is allowed to say no. It says no by filling in an error field on its reply. That reply travels back up a short chain of function calls, and at the top of that chain, nobody looked at the field.

genesis.json
   |
   v
Handshaker.ReplayBlocks          <- the top of the chain
   |
   v
InitChainSync                    <- pass-through, always reports success
   |
   v
baseapp.InitChain                <- reads the error field, only to skip its own bookkeeping
   |
   v
gnoland InitChainer              <- decides the genesis is bad, sets the error field

Because the top of the chain ignored the field, a node whose genesis had been rejected went right on to save its genesis records and commit its first block. It ran, on empty state. A real node did log the reason, but it logged it and carried on, and the fork test ran with logging switched off by default, so there the only symptom was an app hash that looked wrong to anyone who knew what to compare it against.

The change is to look at the field, and stop.

Why the tool inherited the blind spot

The fork test had a backstop of its own. It counted how many transactions the genesis said it should replay, counted how many actually ran, and complained if the second number was smaller. That catches a replay that stops halfway.

It does not catch a genesis that is rejected before replay even begins, because then both numbers are the number the genesis started with. And in the case that motivated this work the genesis carried no transactions at all, so the check compared zero against zero and waved it through.

The two failures the backstop was named after, a mismatched starting height and a malformed signer record, are both raised before the transaction loop. Neither has anything to do with how many transactions the file holds.

Replay stops halfway through the transactions Tool's own backstop catches it Refuses to start, upstream

The last two rows are the interesting ones. Both were already broken before this change; they were just broken quietly. Making them loud is the point, and it is also where the surprises live, because "the loader does not accept this shape" turns out to cover more genesis files than anyone had counted.

Concepts

ABCI
The interface between the consensus engine and the application. InitChain is the call that hands the application its genesis.
Handshake
The step at node startup where consensus and the application agree on how far along each of them is. On a brand new chain that means sending the genesis across and saving the result.
Genesis transaction
A transaction written into the genesis file itself and executed before any block exists. A chain upgrade genesis is mostly these.
App hash
A fingerprint of the application's state, agreed on per block. Before this change, an unexpected one was the sign that genesis loading had never happened on a node that started anyway.
Starting height
A chain upgrade does not start at block 1; it continues the numbering of the chain it replaces. The height is written in two places in the file, and they have to agree.
