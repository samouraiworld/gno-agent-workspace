## Summary

The harness is a separate Go module: each `expected/<slice>.yaml` names a rule and a vulnerable/fixed fixture pair, and `internal/auditpattern` runs `gno test` on each fixture plus a line-based pattern rule, asserting the vulnerable side flags and the fixed side stays clean. The rules are deliberately heuristic (text over a gofmt-normalized, string/comment-blanked view of the source). The reference realm and the security-guide additions ([§5.8](https://github.com/gnolang/gno/blob/e8281bcbe/docs/resources/gno-security-guide.md?plain=1#L330) `OriginCaller`-as-auth, §5.9 raw text in `Render`) are the builder-facing payload; the storage docs reframe "prefer AVL" into "choose storage by access pattern." `AGENTS.md` adds a Security trigger table mapping seven vulnerability families to "what to look for," plus a pointer to run the harness against unfamiliar realm code. The PR is fully additive and touches no consensus, VM, or stdlib runtime code, so blast radius is confined to docs, examples, and a dev tool. The two remaining findings are integrity gaps in the harness-as-contract claim, not runtime risk.

## Glossary

- crossing function — a `.gno` function whose first parameter is `cur realm`; entered via `f(cross, ...)`, which mints a fresh `cur` and shifts `PreviousRealm`.
- `cur.IsCurrent()` — true only when `cur` is the topmost crossing frame's live realm token; at a properly entered crossing function it is true by construction.
- `cur.Previous()` — the realm that crossed into the current frame; reads the captured prev field verbatim, with no IsCurrent validation of its own.
- `IsUserCall()` — pure on the stored `pkgPath`: true only for a direct EOA `maketx call`, false for code realms and ephemeral `maketx run` realms.
- `OriginCaller()` — the transaction signer; same across intermediate realms, so unsafe as an immediate-caller identity. Lives in `chain/runtime/unsafe`.

  <details><summary>details</summary>

  The `interface_realm_param` and `callback_param` slices both flag a `realm` reaching caller-supplied code, which is the real hole. A realm value is a frame-scoped capability, not a bearer token: `cross(rlm)` accepts only while `rlm.IsCurrent()` holds, the token is unforgeable (heap-pointer identity, [`uverse.go:1497-1503`](https://github.com/gnolang/gno/blob/e8281bcbe/gnovm/pkg/gnolang/uverse.go#L1497-L1503)) and non-persistable, so the only escape is handing the live token to code the realm did not audit. But threading `cur` through concrete, statically-imported `/p/` functions is safe and necessary: daokit's interrealm-v2 port ([gnodaokit#64](https://github.com/samouraiworld/gnodaokit/pull/64)) threads it down its Execute chain, and [#5786](https://github.com/gnolang/gno/issues/5786) is the language side of the same pattern. The material shows only the leak; one line, "the danger is a caller-supplied `func`/`interface` value, threading through your own concrete code is fine," keeps a reader from over-correcting. [@moul](https://github.com/gnolang/gno/pull/5835#discussion_r3475424457) accepted this ("not done yet ... I'll add it to the security guide alongside the callback/interface families"). Teaching-completeness gap, not a code defect.
