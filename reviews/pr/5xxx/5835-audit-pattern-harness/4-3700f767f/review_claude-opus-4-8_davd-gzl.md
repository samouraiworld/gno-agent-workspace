## Summary

The harness is a separate Go module: each `expected/<slice>.yaml` names a rule and a vulnerable/fixed fixture pair, and `internal/auditpattern` runs `gno test` on each fixture plus a line-based pattern rule, asserting the vulnerable side flags and the fixed side stays clean. Round 4 tightens two rules, documents them and the report output, comments the fixtures, and links the harness from the quickstart. The PR remains fully additive and touches no consensus, VM, or stdlib runtime code, so blast radius is confined to docs, examples, and a dev tool. The remaining open items are integrity gaps in the harness-as-contract claim (W4/W6/W9), not runtime risk, plus a handful of disclaimed-heuristic and teaching nits carried from round 3.

## Glossary

- crossing function — a `.gno` function whose first parameter is `cur realm`; entered via `f(cross, ...)`, which mints a fresh `cur` and shifts `PreviousRealm`.
- `cur.IsCurrent()` — true only when `cur` is the topmost crossing frame's live realm token; at a properly entered crossing function it is true by construction.
- `cur.Previous()` — the realm that crossed into the current frame.
- `IsUserCall()` — pure on the stored `pkgPath`: true only for a direct EOA `maketx call`, false for code realms and ephemeral `maketx run` realms.
- `OriginCaller()` — the transaction signer; same across intermediate realms, so unsafe as an immediate-caller identity.
