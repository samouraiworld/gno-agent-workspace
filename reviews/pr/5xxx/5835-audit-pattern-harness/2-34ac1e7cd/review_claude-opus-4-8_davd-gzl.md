## Summary

The harness is a separate Go module: each `expected/<slice>.yaml` names a rule and a vulnerable/fixed fixture pair, and `internal/auditpattern` runs `gno test` on each fixture plus a line-based pattern rule, asserting the vulnerable side flags and the fixed side stays clean. The rules are deliberately heuristic (text, not AST), and `README.md` documents the known false-positive/negative classes honestly. The reference realm and the security-guide additions ([§5.8](https://github.com/gnolang/gno/blob/34ac1e7cd/docs/resources/gno-security-guide.md#L330) `OriginCaller`-as-auth, [§5.9](https://github.com/gnolang/gno/blob/34ac1e7cd/docs/resources/gno-security-guide.md#L376) raw text in `Render`) are the builder-facing payload; the storage docs reframe "prefer AVL" into "choose storage by access pattern." Round 2 adds an `AGENTS.md` Security section: a trigger table mapping seven vulnerability families to "what to look for," plus a pointer to run the harness against unfamiliar realm code. The PR is fully additive and touches no consensus, VM, or stdlib runtime code, so blast radius is confined to docs, examples, and a dev tool. The findings below are correctness defects in material whose whole value is being correct, not runtime risk.

|---|---|---|

| `current-guard` | dropping `IsCurrent()` lets a forged token whose `IsCurrent()` is false pass the address check | guard rejects it |
| `payment-user-call` | `IsUser()` accepts an ephemeral `/e/<addr>/run` realm | `IsUserCall()` rejects it |

The realm token cannot be replayed or escalated: `cross(rlm)` accepts only while `rlm.IsCurrent()` holds, so `cross(cur.Previous())` panics `cross: rlm is not the current cur` ([uverse.go:344-348](https://github.com/gnolang/gno/blob/34ac1e7cd/gnovm/pkg/gnolang/uverse.go#L344-L348)). This bounds the `interface-realm-param`/`callback-param` leak to the live synchronous frame.

## Glossary

- crossing function — a `.gno` function whose first parameter is `cur realm`; entered via `f(cross, ...)`, which shifts `PreviousRealm`.
- `cur.IsCurrent()` — true only when `cur` is the topmost crossing frame's live realm token; rejects stale or forged realm values.
- `cur.Previous()` — the realm that crossed into the current frame, i.e. the immediate caller.
- `IsUserCall()` — true only for a direct EOA `maketx call`; false for code realms and for ephemeral `maketx run` (`/e/.../run`) realms.
- `OriginCaller()` — the transaction signer; same across intermediate realms, so unsafe as an immediate-caller identity. Lives in `chain/runtime/unsafe`.
- ephemeral realm — a `gno.land/e/<addr>/run` realm created by `maketx run`; `IsUser()` accepts it, `IsUserCall()` does not.

  <details><summary>details</summary>

  All five scoped scanners (`current_guard`, `payment_user_call`, `render_markdown_escape`, `interface_realm_param`, `render_map_iteration`) advance `braceDepth` with `strings.Count(line, "{")` / `"}"` over the raw line, blind to strings and comments. A `}` inside a string literal closes the function scope early; a `{` inside one holds it open past the real end. README disclaims false positives, so this is precision polish, not a defect: stripping string and line-comment spans before counting braces removes the whole class. Failing test to implement against: [`tests/brace_counting_test.go`](tests/brace_counting_test.go) — the two `NoFalsePositive` cases fail at 34ac1e7cd and pass once strings and comments are stripped, while `StillCatchesRealBug` keeps the genuinely unguarded case flagged so the fix cannot over-blind the rule.

The README's "Known limitations" section documents per-rule caveats for `current_guard`, `payment_user_call`, and `render_markdown_escape`, but has no subsection for `origin_caller_auth` or `exported_pointer_leak`, so two of these are undocumented. All polish on a disclaimed heuristic tool: tighten the heuristics (skip `New*`/`Make*` constructors, parse the actual `Render` parameter name, scan struct fields) or extend the README caveats.

- [`interface-realm-param/vulnerable/hook.gno:4`](https://github.com/gnolang/gno/blob/34ac1e7cd/misc/audit-pattern-harness/fixtures/interface-realm-param/vulnerable/hook.gno#L4) · [↗](../../../../../.worktrees/gno-review-5835/misc/audit-pattern-harness/fixtures/interface-realm-param/vulnerable/hook.gno#L4) — the harness teaches the realm-into-caller-supplied-code leak but not its safe counterpart, threading `cur` through concrete `/p/` code, so a reader may conclude "never give a realm to `/p/`."
  <details><summary>details</summary>

  The `interface_realm_param` and `callback_param` slices both flag a `realm` reaching caller-supplied code, which is the real hole. A realm value is a frame-scoped capability, not a bearer token: `cross(rlm)` accepts only while `rlm.IsCurrent()` holds, the token is unforgeable (heap-pointer identity, [`uverse.go:344-348`](https://github.com/gnolang/gno/blob/34ac1e7cd/gnovm/pkg/gnolang/uverse.go#L344-L348)) and non-persistable, so the only escape is handing the live token to code the realm did not audit. Verified by run: `cross(cur.Previous())` panics `cross: rlm is not the current cur (stale capture or sibling frame)`, so an ancestor or stale token cannot act. But threading `cur` through concrete, statically-imported `/p/` functions is safe and necessary: daokit's interrealm-v2 port ([gnodaokit#64](https://github.com/samouraiworld/gnodaokit/pull/64)) threads it down its Execute chain, and [#5786](https://github.com/gnolang/gno/issues/5786) is the language side of the same pattern. The material shows only the leak; one line, "the danger is a caller-supplied `func`/`interface` value, threading through your own concrete code is fine," keeps a reader from over-correcting. [@moul](https://github.com/gnolang/gno/pull/5835#discussion_r3443319318) already noted the guide "should help choosing the appropriate one depending on the context"; the safe-vs-leak boundary is one such choice. Teaching-completeness gap, not a code defect.
  </details>
