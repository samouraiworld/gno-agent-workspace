What this PR is

Every realm (smart contract) on gno.land stores data on chain: counters, boards, user records, even functions with their captured variables. Until now there was no friendly way to look at that stored data. The State Explorer adds one: every realm gets a "State" tab in gnoweb where its variables and objects become browsable web pages.

A first version of the explorer rendered everything in the browser with TypeScript. This PR moves the rendering to the server, into gnoweb's Go template pipeline. The chain query endpoints and the JSON API stay the same; what changes is who turns the raw data into HTML. Server rendering means every view has a stable URL you can bookmark, share in a chat, or screenshot, and the pages work without JavaScript.

Concepts

Realm — a Gno smart contract with persistent state, addressed by a path like /r/demo/boards.

Realm state — the realm's top-level variables and every object reachable from them, stored on chain as a graph of objects.

ObjectID — the address of one stored object, a realm hash plus a counter, e.g. 0fb476...77f7:16. Objects point at each other by ObjectID.

Amino JSON — the chain's serialization format. Values carry an explicit @type tag so a decoder knows what it is looking at.

htmx fragment — a small HTML snippet the page fetches on demand (for example when a card scrolls into view) and swaps into place, without a full page reload.

How a page is built

Browser                gnoweb (feature/state)                 gno.land node
   |                          |                                     |
   |  GET /r/demo/boards$state|                                     |
   |------------------------->|  validate inputs (oid, search,      |
   |                          |  offset, limit; reject bad ones)    |
   |                          |                                     |
   |                          |  2 RPCs, in parallel:               |
   |                          |---- vm/qpkg_json  (the state) ----->|
   |                          |---- vm/qdoc       (doc comments) -->|
   |                          |                                     |
   |                          |  decode amino JSON (walker),        |
   |                          |  render Go templates                |
   |  full HTML page          |                                     |
   |<-------------------------|                                     |
   |                          |                                     |
   |  ...user scrolls...      |                                     |
   |  GET $state&frag=node    |                                     |
   |------------------------->|---- 1 RPC for that object --------->|
   |  HTML fragment           |                                     |
   |<-------------------------|                                     |

The initial render always costs two chain queries, no matter how large the realm is. Nested objects load later, one small fragment at a time, as they scroll into view. That keeps the first paint fast and prevents one page view from hammering the node.

URL anatomy

gnoweb separates its own parameters from the realm's with a $: everything after it belongs to gnoweb, not to the realm's render function.

URLWhat you get
/r/demo/boards$stateThe realm's top-level declarations as cards, with search, kind filters (State / Code / Types) and pagination.
/r/demo/boards$state&oid=<hash>:16A focused page for one stored object, with its owner, size and type in a sidebar.
/r/demo/boards$state&view=treeThe same data as a dense, collapsible tree instead of cards.
/r/demo/boards$state&jsonThe raw chain data as JSON, paginated, for scripts and external tools.

The JSON API

External consumers (block explorers, IDE plugins, scripts, AI agents) read the same data the pages do:

GET /r/demo/boards$state&json
{
  "pkg_path": "/r/demo/boards",
  "total": 214, "offset": 0, "limit": 500,
  "names":  ["gBoards", "gPerms", ...],
  "values": [ { "T": {"@type": "..."}, "V": {"@type": "/gno.RefValue",
                "ObjectID": "0fb476...77f7:16", ... } }, ... ]
}

References between objects appear as ObjectIDs; following one is another request with &oid=. Errors come back as proper HTTP status codes with a small {"error": "..."} body, so a script can tell "bad request" from "not found" without parsing HTML.

Built-in protections

The explorer reads attacker-visible input (URLs) and chain data of arbitrary shape, so the PR bounds everything:

  • Every URL parameter is validated before any chain query: object IDs, search text, pagination numbers.
  • Chain responses larger than 8 MiB are rejected before decoding.
  • At most 64 chain queries can be in flight at once, process-wide, so a burst of web traffic cannot overload the node.
  • A per-IP rate limiter caps request rates; decode panics on malformed chain data are caught and turned into clean errors instead of crashing the server.
  • Deeply nested or oversized state is cut off at fixed bounds (children per node, decode depth) so a hostile realm cannot stall rendering.
