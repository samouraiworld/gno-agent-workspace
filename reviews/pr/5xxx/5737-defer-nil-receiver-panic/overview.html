Call-time dispatch for interface-bound method values

Plain-language overview of gnolang/gno PR #5737 · generated by claude-opus-4-8

gnovm · matches Go's timing for when a method value picks its concrete method and copies its receiver

A method value is what you get when you write x.Get without the call parentheses. It is a callable you can store, pass, or defer, and it remembers which receiver it came from. The question this PR settles: when does that memory get fixed — the moment you write x.Get, or the moment you finally call it?

The idea in one screen

Concrete receiver — c := &T{1}; g := c.Get. Go fixes everything at the bind: it picks T.Get and snapshots the receiver right there. A later c.x = 2 does not reach g.

Interface receiver — var i I = p; h := i.Get. Go waits until the call. It looks at what i holds at call time, picks the concrete method then, and derefs the pointer then. A p.x = 2 made after the bind but before the call is seen.

GnoVM used to fix the interface case eagerly too, like the concrete case. That is the bug: it diverged from Go on nil-panic timing, receiver snapshots, embedded promotion, field re-reads, and dynamic re-dispatch, and it crashed on defer i.M().

Two moments

Every method value has a bind moment and a call moment. The fix moves interface work from the first to the second.

Concrete c.Get — eager
bindpick method T.Get
bindsnapshot receiver T{1}
bindnil check runs now
calljust invoke the snapshot
Interface i.Get — lazy (this PR)
bindsave the operand + selector name
callread what the interface holds now
callpick method, deref, snapshot
callnil check runs now

In GnoVM the lazy bind is a BoundMethodValue with Func == nil; the saved operand lives in Receiver, the selector in Method, and the bind-site package in MethodPkg. resolveLazyBound does the call-time walk.

Storing an interface method value is a persisted-value shape change, so the fix is a hard fork: it ships on fresh genesis or a coordinated upgrade.

Concepts

method value — the expression x.M without calling it; a first-class callable that carries its receiver.

eager vs lazy bind — eager fixes the concrete method and receiver when the value is formed; lazy defers both to the call.

promoted method — a method reached through an embedded field; which one wins can depend on the value present at call time.

dynamic re-dispatch — if the interface field is reassigned between bind and call, a lazy value calls the new target.

hard fork — a change to a persisted value's wire format; old state still decodes, but block bytes, hashes, and gas differ.
