**Verdict: APPROVE** — 15 surgical fixes targeting real bugs (SaveVersion swallowing DB errors, idempotent-replay corruption, iterator-vs-prune race, first-version orphan leak, eager value-key copy on updates) with matching regression tests; all bptree and store/bptree suites pass. Two doc inconsistencies (README contradicts itself, prune flush-threshold default mismatch — both round-1 findings, unresolved) plus one new design asymmetry to flag (`ImmutableTree.Get` is unprotected against pruning while `ImmutableTree.Iterator` is).

## Summary

15 commits, one finding per commit, on top of the bptree feature branch. The headline correctness fixes are: (1) `SaveVersion` was treating a transient `db.Has` failure as "version doesn't exist" and would proceed to overwrite the persisted root — fixed by an error-propagating `versionExistsE` variant. (2) On a deterministic block replay that hit the "version already exists with same hash" path, session state survived the early-return, and a subsequent `Rollback` would `DeleteValueDirect` value-keys colliding with live persisted entries, wiping real data — fixed by clearing session counters on the idempotent branch. (3) `newIterator` hard-coded `version=0`, so `incrVersionReaders` never fired and `PruneVersionsTo` could delete nodes a live iterator was walking — fixed by routing `imm.version` through the constructor. (4) `pruneVersion` only consumed `orphans[nextV]`, leaking any values listed in `orphans[first]`; the invariant that `orphans[first]` is always empty is locked in by a new test (`TestOrphans_FirstVersionEmpty`), and the defensive double-consume is free (`LoadOrphans` returns empty on missing). Performance: `saveNode` no longer force-loads ~H×(B-1) unchanged sibling nodes per `SaveVersion`, and `treeInsert`'s defensive key copy is deferred from every call to actual-store branches only. Fail-fast guards reject corrupt on-disk payloads (`ReadNode` trailing bytes, `Serialize` nil valueKey/child-ref) and adversarial sizes (`MaxKeyLen` on `Set`/`Import`).

## Glossary

- `vk` / `valueKey`: 12-byte `(version, nonce)` identifier used as the DB key for an out-of-line value (see [`README.md` L37-43](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/README.md#L37-L43) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/README.md#L37-L43)).
- `sessionValues`: in-memory list of `vk`s allocated in the current working session; consumed by `Rollback` (delete-direct) or cleared by `SaveVersion`.
- `versionOrphans` / `orphans[V]`: persisted list of `vk`s displaced WHEN version V was saved (owning version `<V`); consumed during `pruneVersion`.
- Tier 1 / Tier 2: intra-version orphans (eager delete) vs cross-version orphans (deferred to prune). See [`mutable_tree.go:162-176`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/mutable_tree.go#L162-L176) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/mutable_tree.go#L162-L176).
- COW: copy-on-write; every mutation clones the root-to-leaf path.
- Idempotent save: `SaveVersion(v)` when v already exists in the DB with the same hash — the typical deterministic-replay-after-crash case.

## Fix

Per-commit shape: read a bug surface, write a narrow fix, ship a regression test that fails without the fix. The PR does not introduce new public API beyond `MaxKeyLen` / `ErrKeyTooLong` and the (intentionally unexported) `versionExistsE`. The main structural addition is `nodeDB.cleanupCrashedSessionValues` ([`nodedb.go:379-419`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/nodedb.go#L379-L419) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/nodedb.go#L379-L419)) wired into `MutableTree.Load` ([`mutable_tree.go:396-411`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/mutable_tree.go#L396-L411) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/mutable_tree.go#L396-L411)): a contiguous-suffix scan above `latestVersion` that deletes values from a session that died before `SaveVersion`. Cost is O(orphans), with zero overhead on clean shutdown. `ImmutableTree` gains an `ndb` field set by `GetImmutable`, threaded through to `Iterator` so the iterator can `incr/decrVersionReaders` against the snapshot's own version rather than the previously hard-coded `version=0`.

The `if flushThreshold <= 0 { flushThreshold = 4 * 1024 * 1024 }` branch ([`prune.go:37-39`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/prune.go#L37-L39) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/prune.go#L37-L39)) is dead. The test passes by coincidence (100 KiB is also enough for 10 trivial versions). Two consistent options: drop the fallback and document the 100 KiB default; or raise `DefaultOptions().FlushThreshold` to 4 MiB to match the prune-internal expectation (and adjust the per-block save Commit, which today flushes nodes within the same 100 KiB envelope). - **[ImmutableTree.Get unprotected vs prune]** [`immutable_tree.go:39-48`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/immutable_tree.go#L39-L48) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/immutable_tree.go#L39-L48) — `ImmutableTree.Iterator` now registers a version reader (the fix of `ccbbeb66b`), but `ImmutableTree.Get`/`Has`/`GetByIndex`/`GetWithIndex` do not — leaving point reads on historical heights vulnerable to the same race the iterator fix closed.
  <details><summary>details</summary>

  `Store.GetImmutable(v)` returns a store backed by an `ImmutableTree` ([`store/bptree/store.go:71-92`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/store/bptree/store.go#L71-L92) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/store/bptree/store.go#L71-L92)). Cosmos-SDK style query handling typically holds an immutable store across multiple `Get` calls while block-processing schedules `DeleteVersionsTo`. With the iterator fix, an open `Iterator(v)` correctly blocks pruning of v. But a long-lived `Get`/`Has` series on `imm.GetImmutable(v)` does not register a reader — `getChild` lazy-loads from DB and will panic on a missing node ([`node.go:73-76`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/node.go#L73-L76) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/node.go#L73-L76)) if v was pruned mid-query.

  The asymmetry is what makes this a Warning rather than a Nit: the PR explicitly fixed the iterator side and documented the rationale (`PruneVersionsTo` must not race with active readers), but only half the read surface is now covered. The cleanest fix is to register the reader at `GetImmutable` (held for the lifetime of the `ImmutableTree`) rather than at each `Iterator` call site — that way every read path on the immutable snapshot inherits the protection, and the existing `Iterator` registration becomes a no-op double-incr (or is removed). Alternative: document explicitly that `ImmutableTree.Get`-style calls are unsafe across prune boundaries and rely on the caller to wrap them in an `Iterator(start, end=nil)` (awkward but explicit).

  Note this is largely a pre-existing gap not introduced by this PR — but the iterator fix here is what surfaces it.
  </details>

- **[divergent-replay duplicates leak forever]** [`mutable_tree.go:273-281`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/mutable_tree.go#L273-L281) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/mutable_tree.go#L273-L281) — The idempotent-save fix correctly handles the collide-by-nonce-order path, but the documented "divergent replay" caveat produces uncollected duplicates that `cleanupCrashedSessionValues` cannot reach.
  <details><summary>details</summary>

  The fix clears session counters on idempotent-save without `DeleteValueDirect`'ing the session vks — correct for the collide case (those are the live entries). The comment then acknowledges that if the replay's allocation order diverges (e.g., transient Set/Remove pairs that cancel out), the replay's vks have version == replayed-version but different nonces; they don't collide with the persisted vks. Those replay vks become unreferenced DB entries with `version <= latestVersion`. They are not caught by `cleanupCrashedSessionValues` ([`nodedb.go:379-419`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/nodedb.go#L379-L419) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/nodedb.go#L379-L419)) because that scan is bounded to `version > latestVersion`.

  The README claims "values are never garbage collected — dead values after pruning are harmless noise" ([`README.md:46-47`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/README.md#L46-L47) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/README.md#L46-L47)), so this aligns with the design intent. But "harmless noise" assumes those values stay bounded relative to live state. A divergent-replay path that fires every block (e.g., on a node configured for aggressive replay) accumulates unboundedly because pruning the underlying version doesn't catch them either: `orphans[V]` lists values displaced AT V, not duplicates allocated DURING V. Fix: in the idempotent-save branch, walk `t.sessionValues` and bail if any vk is not in the persisted set (a definitive signal of divergence), then either log + leak, or attempt cleanup of the duplicates only. At minimum, surface a counter so the leak is observable, and update the README's "harmless noise" claim to note this specific corner case.
  </details>

## Suggestions

- [`nodedb.go:393-411`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/nodedb.go#L393-L411) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/nodedb.go#L393-L411) — `cleanupCrashedSessionValues` materializes all orphan keys into `toDelete` then deletes one-by-one via direct `db.Delete`. For a pathological crash that left millions of session vks behind, this is both unbounded memory and N round-trips. Threshold-based batching (the same pattern wired up in `PruneVersionsTo` by commit `4dd84b894`) would cap both. Crash recovery is rare and one-shot, so this is low priority, but the symmetry is appealing — and it's the same `flushThreshold` knob, so cost is small.

- [`prune.go:50-58`](https://github.com/gnolang/gno/blob/ee34a2dad/tm2/pkg/bptree/prune.go#L50-L58) · [↗](../../../../../.worktrees/gno-review-5591/tm2/pkg/bptree/prune.go#L50-L58) — `findNextVersion` does `O(latest - v)` `db.Has` calls per call, and `PruneVersionsTo`'s outer loop does it per surviving version. For dense version sequences this is fine; for sparse sequences (e.g., after multiple non-contiguous prunes) it becomes quadratic in `latest`. Not new in this PR, but the dual-orphan-consume change here is one place to consider caching the result of `discoverVersions` for the prune loop's duration.

- README — `MaxKeyLen` is now a stable external constant exported alongside `ErrKeyTooLong`. Mention it in the "Out-of-line values" or "Design choices" section so users hitting `ErrKeyTooLong` find the limit documented.
