**TL;DR:** When the chain type-checks a submitted package, the accept/reject decision and the rejection text both feed consensus, so every node must agree. This PR makes them agree regardless of which Go toolchain each node's binary was built with: it pins the accepted Go language version, and it replaces the toolchain-dependent error text in the hashed transaction result with a fixed placeholder while keeping the full detail on the unhashed log the user sees.

**Verdict: REQUEST CHANGES** — the error-coarsening half is complete and well tested, but the `GoVersion` pin does not achieve the invariant it states: one `//go:build` comment line in a submitted file overrides it, and the verdict still resolves against the toolchain the validator was built with. Two binaries built from this commit with Go versions `go.mod` admits reach opposite verdicts on the same package, one deploying it and one rejecting it.

## Summary

Two independent ways two honest validators could fork on the same submitted package, no attacker needed. First, the consensus type-checker built its `go/types` config with no `GoVersion`, so version-gated syntax (e.g. `for range 10`, a go1.22 feature) was accepted or rejected according to whichever Go the validator binary was compiled with. Second, on rejection the raw `go/types`/`go/parser` diagnostic strings were amino-encoded into `TypeCheckError.Errors` and hashed into the block's results hash, and that text drifts in wording, count, and order across toolchains, so two nodes correctly rejecting the same package could still commit different result hashes. The fix pins `GoVersion` to `go1.18` and collapses `TypeCheckError` to an empty sentinel, moving the full diagnostics onto the unhashed `Result.Log` trace.

The second axis is closed. The first is not: `go/types` resolves a per-file `//go:build go1.N` line *above* the config pin, and gates that file version against the version of the Go toolchain that compiled the binary. Both behaviours survive the pin, and package bodies are attacker-supplied.

```
submitted file: "//go:build go1.26\n\npackage p\nfunc Add(a,b int) int { return a+b }"
                         │
        ┌────────────────┴────────────────┐
   validator A                       validator B
   built w/ go1.26                   built w/ go1.25
        │                                 │
   file ver go1.26 <= go_current      file ver go1.26 > go_current
        │                                 │
   ACCEPT → package deployed          REJECT → TypeCheckError
        │                                 │
   app hash H1                        app hash H2         ← chain halts
```

## Examples

| A user submits | Before | After |
|---|---|---|
| `for range 10` (range-over-int, go1.22) | accepted on a go1.22+ build, rejected on older | rejected on every build |
| `//go:build go1.22` + `for range 10` | accepted everywhere | still accepted everywhere: the pin is overridden |
| `//go:build go1.26` + ordinary valid code | build-dependent | build-dependent: deployed on a go1.26 build, rejected on a go1.25 build |
| any type-check failure | raw go/types text hashed into the result | fixed sentinel hashed; full text stays on the unhashed log |

## Glossary

- type-check: go/types-based validation of gno source (`TypeCheckMemPackage`), distinct from preprocessing.
- results hash: per-block Merkle commitment to every tx's `ABCIResult`; toolchain-dependent bytes in `ABCIResult.Error` fork the chain.
- app hash: per-block commitment to application state; divergence halts the chain.
- gnobuiltins: synthetic packages injected only for type-checking, never run on chain.
- Amino: gno's deterministic serialization codec; here it encodes the hashed error.
- preprocess: the static pass that resolves names, types, and blocks before execution.

## Critical (must fix)

- **[submitter picks the language version; verdict still depends on the build]** [`gotypecheck.go:190`](https://github.com/gnolang/gno/blob/7fc5ec06a/gnovm/pkg/gnolang/gotypecheck.go#L190) · [↗](../../../../../.worktrees/gno-review-5893/gnovm/pkg/gnolang/gotypecheck.go#L190) — a `//go:build go1.N` line in a submitted file overrides the pinned `GoVersion`, and the resulting file version is gated against the Go toolchain the validator binary was built with, so the accept/reject verdict is still a function of the build.
  <details><summary>details</summary>

  `go/types` resolves each file's language version in `initFiles`: when a file carries a `//go:build go1.N` line it sets that file's version to `max(fileVersion, go1.21)`, with no gate on `Config.GoVersion`, then rejects the file outright when `fileVersion > go_current`, where `go_current` is `fmt.Sprintf("go1.%d", goversion.Version)` — a constant baked in at compile time. `.gno` files reach `go/types` through stdlib `go/parser`, which populates `ast.File.GoVersion` from `//go:build` lines, and `prepareGoGno0p9` does not strip them. Both consequences are live: `//go:build go1.22` + `for range 10`, `//go:build go1.21` + `min(1, 2)` and `//go:build go1.23` + range-over-func all type-check clean under the go1.18 pin, and a file tagged above the builder's toolchain is rejected only on the older build. `go.mod` declares `go 1.25.9` with no `toolchain` directive, so go1.25 and go1.26 builds are both in scope today. Built from this commit, a go1.26.5 binary accepts and deploys `//go:build go1.26` + ordinary valid code while a go1.25.9 binary rejects it with the `TypeCheckError` sentinel: the two disagree on state, not merely on the results hash. Nothing here is a regression, and the PR is a strict improvement on its own axis: an isolated `go/types` harness gives byte-identical verdicts for tagged files with and without the pin, and the toolchain gating predates the PR untouched. The gap is that the gate covers untagged files only, while the comment and the PR body state the verdict is now build-independent for every submitted package. So the invariant the pin states is not achieved, and [`gotypecheck.go:187-189`](https://github.com/gnolang/gno/blob/7fc5ec06a/gnovm/pkg/gnolang/gotypecheck.go#L187-L189) · [↗](../../../../../.worktrees/gno-review-5893/gnovm/pkg/gnolang/gotypecheck.go#L187)'s claim that the pin rejects what Gno can't run "here rather than downstream" is falsified by the tagged case, which reaches the preprocessor and dies there with `got BigintKind`. Fix: clear `ast.File.GoVersion` on every parsed `.gno` file in [`GoParseMemPackage`](https://github.com/gnolang/gno/blob/7fc5ec06a/gnovm/pkg/gnolang/gotypecheck.go#L587) · [↗](../../../../../.worktrees/gno-review-5893/gnovm/pkg/gnolang/gotypecheck.go#L587), next to the existing `prepareGoGno0p9` pass, so no per-file version can be set from inside a submitted package; build constraints have no meaning in Gno, and blanking the field closes both halves at once. Ready-to-add tests: [`gotypecheck_buildtag_pin_test.go`](tests/gotypecheck_buildtag_pin_test.go) and [`keeper_buildtag_addpkg_test.go`](tests/keeper_buildtag_addpkg_test.go); [repro](comment_claude-opus-4-8.md).
  </details>

## Open questions

- At a fixed `GoVersion`, `go/types` acceptance can still drift across Go releases through implementation fixes on non-version-gated code. Round 1 filed this as the residual; the version-gated part of it turned out to be concrete and is now the Critical above. What remains is genuinely open-ended and only fully closed by vendoring a pinned `go/types`, which is large. Worth noting that the PR's second half already contains the blast radius: under the error coarsening, a toolchain that merely words or counts diagnostics differently no longer changes the hashed bytes, so drift has to flip accept/reject, not just phrasing, to fork. Not posted: no change is asked in this PR.
- Only the type-check path was coarsened. A package that passes type-check and dies in the preprocessor has that preprocessor text hashed into the result as a `StringError` (observed: `range iteration requires map, string, array, slice, or pointer to array; got BigintKind`). That text is generated by gno's own code rather than the host toolchain, so it is deterministic across builds and forks nothing today. Flagged only because the Critical above is what routes tagged packages down that path. Not posted.
