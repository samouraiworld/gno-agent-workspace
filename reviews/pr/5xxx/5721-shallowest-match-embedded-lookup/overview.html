Shallowest-match wins for embedded field/method lookup

Plain-language overview of gnolang/gno PR #5721 · generated by claude-opus-4-8

gnovm · resolves x.f on embedded types by depth, matching Go's spec, instead of by field order

When a struct embeds other types, a field or method of an embedded type can be reached directly on the outer struct. That is promotion. If the same name lives at more than one place in the embedding graph, which one does x.f pick? Go's rule is simple: the shallowest one wins, and a tie at that depth is an error. GnoVM used to walk the graph depth-first in field order, so it could pick a deeper match, silently swallow a same-depth clash, or report a reachable name as missing. This PR switches the walk to breadth-first so it matches Go.

Depth, in one picture

Depth is how many embedding hops separate the outer struct from the type that declares the name. A field of a directly embedded type is depth 1; go one more level in and it is depth 2.

Shallowest wins — among all types that declare the name, the one fewest hops away is chosen.

Tie is ambiguous — if two different types declare the name at the same shallowest depth, the selector is a compile error, not a silent pick.

A deeper unique match does not rescue a shallower tie — once the shallowest depth is ambiguous, the lookup stops; it does not fall through to a lone deeper candidate.

The selector rules run before the VM through the go/types type-check that a deploy already passes, so no realm that reached the VM was relying on the old permissive behavior. The fix aligns the VM with the gate in front of it.

Concepts

embedding — a struct field written as just a type name; its fields and methods are reachable on the outer struct.

promoted field / method — a name reached through embedding rather than declared directly.

selector — the x.f expression the compiler resolves by walking the embedding graph.

ambiguous selector — two providers of the name at the same shallowest depth; a compile error in Go and now in GnoVM.

breadth-first (BFS) — visit the graph level by level, so the shallowest provider is found before any deeper one.
