## Summary

Lands the entire `tm2/pkg/bft/privval/upstream/` transport layer — `SignerListenerEndpoint` + `SignerClient` + `RetrySignerClient` (cometbft v0.39.1 ports) and a parallel `SecretConnection` (cometbft v0.34.34 port with Merlin-bound STS handshake), kept as a separate copy so the chain-p2p `SecretConnection` stays byte-stable. ## Glossary

- `SignerListenerEndpoint` — validator-side listener (cometbft v0.39.1). Owns the conn; sends pings every (timeoutReadWrite·2/3); reconnects on drop.
- `SignerClient` — implements tm2's `types.PrivValidator` over the endpoint. Holds the cached identity (`cachedPubKey`) anchored at `Init()`.
- `RetrySignerClient` — wraps `SignerClient`; sleeps and retries on transient errors; passes `WrappedRemoteSignerError` through immediately (signer-side refusal must not be retried).
- `verifyIdentity` — re-fetches the signer's pubkey after a reconnect and compares it to `cachedPubKey`; refuses to sign for a swapped signer.
- `connGen` — monotonic counter bumped every time a fresh conn is installed. Used to gate re-verify.
- `upstream/SecretConnection` — Merlin-transcript-bound STS handshake, byte-compatible with cometbft v0.34 (the chain-p2p `tm2/pkg/p2p/conn/SecretConnection` is intentionally wire-different and stays untouched).
- `AuthSigMessage` — protobuf `{ PublicKey pub_key = 1; bytes sig = 2 }` exchanged inside the AEAD-protected channel during the SecretConnection handshake.
- `upstreampb` — protoc-generated Go types vendored from upstream Tendermint v0.34's `types.proto` / `canonical.proto` / `privval/types.proto`. Used for actual wire I/O (the amino-encoded mirrors in `upstream.{Vote,Proposal,...}` are documentation/byte-compat-verification only).

## Fix

Before: tm2 had a single privval implementation (`gnokms`, in `tm2/pkg/bft/privval/signer/remote/`) whose wire format diverges from upstream Tendermint v0.34 in two load-bearing places — the canonical sign-bytes ([`tm2/pkg/bft/types/canonical.go`](https://github.com/gnolang/gno/blob/235c1eb72/tm2/pkg/bft/types/canonical.go) · [↗](../../../../../.worktrees/gno-review-5717/tm2/pkg/bft/types/canonical.go), addressed in #5716) and the SecretConnection AuthSigMessage shape (a 2-byte amino vs protobuf-oneof shift, [`tm2/pkg/p2p/conn/secret_connection.go`](https://github.com/gnolang/gno/blob/235c1eb72/tm2/pkg/p2p/conn/secret_connection.go) · [↗](../../../../../.worktrees/gno-review-5717/tm2/pkg/p2p/conn/secret_connection.go)). Result: `tmkms` and other upstream-protocol KMS cannot connect to or sign-verify against a gno.land validator.

After: parallel `tm2/pkg/bft/privval/upstream/` package holding a `SignerListenerEndpoint` + `SignerClient` + `RetrySignerClient` + a separate `SecretConnection` copy that speaks upstream's v0.34 STS handshake. The chain-p2p SecretConnection is unchanged (a doc-pinned canary test in [`tm2/pkg/p2p/conn/secret_connection_compat_test.go`](https://github.com/gnolang/gno/blob/235c1eb72/tm2/pkg/p2p/conn/secret_connection_compat_test.go) · [↗](../../../../../.worktrees/gno-review-5717/tm2/pkg/p2p/conn/secret_connection_compat_test.go) asserts the 2-byte divergence still exists, so a future blanket edit can't "fix" it without a hard-fork review). The upstream-compat listener wraps each inbound TCP dial with the new `SecretConnection` and gates on a pubkey allowlist (defaulting to fail-open, like gnokms) before handing the verified conn to `SignerListenerEndpoint`. UDS Accept skips SecretConnection (kernel-level isolation, same trade-off as cometbft).

Six commits at the head of the branch (`fix(privval/upstream): ...`) close defense-in-depth gaps surfaced by an adversarial bug-hunt loop: see [`bug_fixes_test.go`](https://github.com/gnolang/gno/blob/235c1eb72/tm2/pkg/bft/privval/upstream/bug_fixes_test.go) · [↗](../../../../../.worktrees/gno-review-5717/tm2/pkg/bft/privval/upstream/bug_fixes_test.go) for the full set with one regression test per bug. The two structurally interesting ones:

- **Bug #6 (verify TOCTOU)**: `DropConnection` clears `se.conn` but does NOT bump `connGen`. The pre-fix `SignVote` ran `verify → send` and `send`'s `ensureConnection` was what bumped `connGen` on a re-dial — so `verify` short-circuited on stale gen before the fresh peer's conn was installed. Fix: split `SendRequestLocked` into `EnsureConnectionLocked + SendRequestOnConnLocked` so `SignVote` runs `ensure → verify → send` ([`signer_client.go:179-184`](https://github.com/gnolang/gno/blob/235c1eb72/tm2/pkg/bft/privval/upstream/signer_client.go#L179-L184) · [↗](../../../../../.worktrees/gno-review-5717/tm2/pkg/bft/privval/upstream/signer_client.go#L179-L184)).
- **Bug #1 / #4 (`stopCh`)**: `WaitForConnection` held `instanceMtx` through the wait, so `OnStop`'s `instanceMtx.Lock()` blocked behind it. `BaseService.Quit()` closes only AFTER `OnStop` returns, so adding a `Quit()` select arm would have been a deadlock by another name. Fix: dedicated `stopCh` closed at the START of `OnStop` ([`signer_listener_endpoint.go:53-58`](https://github.com/gnolang/gno/blob/235c1eb72/tm2/pkg/bft/privval/upstream/signer_listener_endpoint.go#L53-L58) · [↗](../../../../../.worktrees/gno-review-5717/tm2/pkg/bft/privval/upstream/signer_listener_endpoint.go#L53-L58)), released `instanceMtx` before the wait, `stopCh` threaded through `signerEndpoint.WaitConnection` so `ensureConnection`-via-`SendRequest` also unblocks promptly ([`signer_endpoint.go:80-90`](https://github.com/gnolang/gno/blob/235c1eb72/tm2/pkg/bft/privval/upstream/signer_endpoint.go#L80-L90) · [↗](../../../../../.worktrees/gno-review-5717/tm2/pkg/bft/privval/upstream/signer_endpoint.go#L80-L90)).
