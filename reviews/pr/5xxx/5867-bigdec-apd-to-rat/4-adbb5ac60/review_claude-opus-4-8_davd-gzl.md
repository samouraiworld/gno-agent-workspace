## Summary
Round 3 rejected two things and both are fixed. [`BigdecValue`](https://github.com/gnolang/gno/blob/adbb5ac60/gnovm/pkg/gnolang/values.go#L133-L136) · [↗](../../../../../.worktrees/gno-review-5867/gnovm/pkg/gnolang/values.go#L133) now carries either an exact `*big.Rat` or a bounded 512-bit `*big.Float`, and [`wrapRatOrPromote`](https://github.com/gnolang/gno/blob/adbb5ac60/gnovm/pkg/gnolang/op_binary.go#L811-L816) · [↗](../../../../../.worktrees/gno-review-5867/gnovm/pkg/gnolang/op_binary.go#L811) switches to the float form when either rational component exceeds 4096 bits, exactly what `go/constant` does at the same threshold. Arithmetic, comparison, negation, and gas metering handle both forms.

The float32 conversion was rewritten to round once at [`big.Float.SetPrec(24)`](https://github.com/gnolang/gno/blob/adbb5ac60/gnovm/pkg/gnolang/values_conversions.go#L1463) · [↗](../../../../../.worktrees/gno-review-5867/gnovm/pkg/gnolang/values_conversions.go#L1463). A 24-bit float equals a normal float32 exactly, so the normal-range double rounding the commit targeted is gone. But a subnormal float32 holds fewer than 24 bits, so pre-rounding to 24 bits and then to the subnormal grid rounds twice, and a value just past a subnormal tie lands on the wrong side. `const s float32 = 0x1.4p-148 + 0x1p-200` is `2` here and `3` in Go.

The second issue is in [`parseBigdecLiteral`](https://github.com/gnolang/gno/blob/adbb5ac60/gnovm/pkg/gnolang/op_binary.go#L862-L876) · [↗](../../../../../.worktrees/gno-review-5867/gnovm/pkg/gnolang/op_binary.go#L862): it calls `big.Rat.SetString` unconditionally, materializing the full integer for a literal like `1e999999` (a 3.3 Mbit numerator, ~16 ms) before `ratOverflows` sees it is too big and throws it away. Arithmetic is bounded because it promotes; literal parsing is not, and this is where `apd` was O(1).

## Glossary
- bigdec: the VM's arbitrary-precision value backing an untyped float constant before it is typed; now a `big.Rat` or a 512-bit `big.Float`.
- preprocess: the static pass that resolves names, types, and blocks and constant-folds expressions before execution; not gas-metered per operation.
- promotion: the automatic switch from exact `big.Rat` to bounded `big.Float` once a value's numerator or denominator exceeds 4096 bits, mirroring `go/constant`.
- subnormal: an IEEE-754 value below the smallest normal magnitude (float32: `< 2^-126 ≈ 1.2e-38`), represented with reduced significand precision.
- signed zero: IEEE-754 keeps distinct `+0` and `-0`; Go constants do not, so any constant that converts to zero yields `+0`.
- filetest: a `.gno` file executed by the VM and asserted against golden directives (`// Output:`).

## Fix
The float32 conversion should round once from the exact value, the way the float64 path already does. [`bigdecToFloat64`](https://github.com/gnolang/gno/blob/adbb5ac60/gnovm/pkg/gnolang/values_conversions.go#L1496-L1506) · [↗](../../../../../.worktrees/gno-review-5867/gnovm/pkg/gnolang/values_conversions.go#L1496) calls `bdv.V.Float64()` / `bdv.F.Float64()` directly, and float64 constants match Go including subnormals. The float32 case should mirror it with `bdv.V.Float32()` / `bdv.F.Float32()` (guarding the zero case), dropping the `SetPrec(24)` intermediate. That gives Go's result for both the subnormal cases and the normal-range case the commit added.

For `parseBigdecLiteral`, parse the float first with `big.ParseFloat` (always cheap and bounded) and build the exact `big.Rat` only when its magnitude fits under 4096 bits; values that would overflow keep the float form and never allocate the giant numerator. In-band literals stay exact, so the headline fix is unaffected.

## Verified
- Both round-3 Criticals are fixed. Built `gnovm/cmd/gno` at adbb5ac60 and ran against Go and master: `println(1e10000 / 1e10000)` is `1` (was panic); `var f float32 = -1e-50` has `math.Float32bits` `0` / `math.Signbit` false (was `-0`). Both match Go. `float9.gno` and `float10.gno`, the two `main / test` failures round 3 named, pass at this head.
- The subnormal float32 divergence, found independently by two lenses and confirmed against Go and master: `const s float32 = 0x1.4p-148 + 0x1p-200` is `2` at adbb5ac60, `3` in Go, `20` on master; `0x1.2p-147 + 0x1p-200` is `4` / `5` / `36`. The current `SetPrec(24)` gives `2`; `bdv.V.Float32()` and `SetPrec(512)` both give Go's `3` and keep the normal-range case at `1065353217`. A 600-case random float32 scan diverges only on subnormals; a 500-case float64 scan (incl. subnormals) is bit-identical.
- Bit-exact parity against real Go across ~1200 untyped-float-constant expressions total: exact rationals, promotion-boundary comparisons and arithmetic (`1e2000>1e1999`, `1e5000-1e5000==0`, `1e4000/1e4000==1`, mixed rat/float), cancellation, hex floats (`0x1p4000`), the six comparison operators, and int conversions. Subnormal float32 is the only category that diverges.
- The double-rounding fix for the normal case: `const x float32 = 1.0 + 0x1p-24 + 0x1p-80` has bits `1065353217`, matching Go and notJoon's [playground case](https://github.com/gnolang/gno/pull/5867#discussion_r3567858061); the old float64-then-narrow path rounded to `1065353216`.
- Amino: both rat and float forms round-trip and re-marshal to a fixed point; `1e-1233` stays rat, `1e-10000` promotes to float. A package-level `const Third = 1.0/3.0` persists as `"1/3"` and reloads from the committed store; `const Huge = 1e5000` persists as `"f:0x…"`. `TestAppHashCrossrealm38` passes with the documented hash, confirming the encoding is deterministic.
- `go/constant`'s precision and threshold match the code's constants: [`prec = 512`](https://github.com/golang/go/blob/go1.25.0/src/go/constant/value.go#L69), [`maxExp = 4 << 10`](https://github.com/golang/go/blob/go1.25.0/src/go/constant/value.go#L351). - The literal-parse cost: 500 `1e999999` consts (~6.5 KB source) fold in 7.9 s at this head; master folds the same in ~0.02 s. `big.ParseFloat` of `1e999999` is ~7 µs vs `big.Rat.SetString`'s ~16 ms. A parse-float-first prototype folds the same 500 in 2.2 ms while keeping in-band literals exact.
