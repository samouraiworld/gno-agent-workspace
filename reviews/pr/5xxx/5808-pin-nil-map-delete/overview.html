delete on a nil map is a no-op

PR #5808 · gnovm · follow-up to #5196 · tests + ADR only, no runtime change · generated by claude-opus-4-8
The Go spec says: "If the map m is nil or does not contain such an element, delete is a no-op." GnoVM used to crash on the nil case: the delete builtin type-asserted the map value to *MapValue without checking for nil first, so a nil map produced an unrecoverable VM abort (interface conversion: Value is nil, not *MapValue). The one-line guard that fixes it already merged in #5196. This PR adds no code: it pins the guard's full behavior with three filetests and records two design decisions in an ADR.

The guarded path

Inside the delete builtin, for a value typed as a map:

key := arg1.Deref()evaluated once, before the type switch
▼
if mapValue == nil → returnthe guard — no-op, nothing hashed
▼
mv := value.(*MapValue)used to panic here on nil
▼
if isReadonly(map) → paniccross-realm write guard
▼
hash key, find entry, delete, mark updatednormal path

The guard sits above the readonly check, but the ordering is invisible: a nil value can never be readonly-tainted (taint is computed from a value's object identity, and nil has none), so a nil map would return false from the readonly check anyway.

Decision 1 — readonly ordering is unobservable

mapreadonly-tainted?delete result
nilimpossible (nil has no object identity)no-op
non-nil, own realmnodeletes
non-nil, external realmyespanics — "cannot delete from readonly tainted map"

Only the non-nil readonly row panics, and no test on this branch asserts that panic: zrealm_map1.gno / zrealm_map3.gno pin successful same-realm deletes since #5747. The new zrealm_mapnil.gno pins the nil cross-realm no-op (the top row, via an external realm).

Decision 2 — unhashable key on a nil map: gno no-ops, gc panics

The one deliberate divergence from the standard Go compiler. With k unhashable (e.g. a slice) and m a nil map:

operationgc (Go compiler)gno
delete(m, k), m nil, k unhashablepanic: hash of unhashable typeno-op
_ = m[k] (read), m nil, k unhashablepanic: hash of unhashable typeno-op pre-existing
delete(m, k), m nil, k hashableno-opno-op

gno reads the spec literally: nil map ⇒ no-op, full stop, before any key hashing. This also matches what gno already did for nil-map reads with unhashable keys (middle row), so delete is now consistent with reads rather than carving out a new special case. Since #5501 (on this branch via the PR's master merge) the slice-key hashing panic is a recoverable runtime error, so gc parity for the pinned slice-key case is implementable; func- and map-typed keys still abort the VM through the plain-panic default, so hash-before-nil-check parity is not yet safe uniformly across key types.

Concepts

filetest — a .gno file under gnovm/tests/files/ with an // Output: block. The VM runs it and the printed output must match exactly.

readonly taint — gno's cross-realm write protection. A value owned by another realm is read-only to the currently executing one; mutating it (including delete) panics. Computed from the value's on-chain object identity, so values with no identity (like a nil map) are never tainted.

ADR — architecture decision record. A short doc capturing what an AI-assisted change decided and why, so the reasoning isn't re-litigated later.
