PR #5794: serve realm pages as markdown via Accept negotiation APPROVE

gnolang/gno#5794 by gfanton · 6 files, +305 −13 · reviewed at 4ab275316 · full review

In one sentence: when a client's Accept header names text/markdown, gnoweb returns the realm's raw Render() output instead of the full HTML page; everyone else sees no change.

A realm's Render() already produces markdown. gnoweb normally pushes it through goldmark and wraps it in the site layout. Agents and scripts want the source, and this PR gives it to them through standard content negotiation: no new endpoint, no query parameter, same URL.

Every GET also carries Vary: Accept, so a shared cache never serves the markdown body to a browser or vice versa. The markdown branch only exists for realm pages and static-markdown aliases; source, help, directory and user views always render HTML for now.

Try it: the negotiation logic

This runs the same decision negotiatesMarkdown() makes in negotiate.go:15-32: split the header on commas, and return markdown on the first part that names text/markdown or text/x-markdown without an explicit q=0. Wildcards never match.

    Open items (none blocking)

    q-preference shortcut: confirm intended

    negotiate.go:15-32

    text/html;q=0.9, text/markdown;q=0.8 returns markdown: any non-zero q wins regardless of how the client ranks the alternatives. A shortcut over RFC 9110 preference ordering, and a test locks it in. Harmless in practice since browsers never send text/markdown, but worth a one-word confirm that it's a choice, not an oversight.

    Add nosniff to the raw write

    handler_http.go:207

    The markdown branch serves realm Render() bytes verbatim, while the HTML branch sanitizes through goldmark. gnoweb sets no nosniff header anywhere today. Risk is low because the branch requires an explicit Accept: text/markdown, but X-Content-Type-Options: nosniff on this one write is cheap insurance on the only raw-bytes surface.
