# Review History

Answer questions about work already reviewed, from `reviews/` on disk. Never re-review to answer a history question. `gh` is allowed only for facts the corpus cannot hold (current PR state, head movement since the review); never to re-derive what is on disk.

**Input:** `$ARGUMENTS`: a PR number, a free-text term, or a mode word below. Empty means `recent 10`.

## Corpus layout

- `reviews/pr/<thousand>xxx/<number>-<slug>/<round>-<short-sha>/` for gno PRs. `overview.html` sits at the PR-dir root; `tests/` inside round dirs.
- `reviews/<repo>/<number>-<slug>/` for non-gno repos. Round dirs are optional: `onboarding-bot` keeps review files directly in the PR dir; `gnodaokit` has round dirs plus a PR-level `tests/`.
- Round files, current naming: `review_<model>_<reviewer>.md` plus `comment_<model>.md`. Older rounds instead hold `<model>_<reviewer>.md` (no `review_` prefix), `review_<model>.md`, tier-split `*__tier*.md` files, bare `comment.md`, or no comment draft at all. A round can hold several reviewer files (parallel lenses); not all carry a verdict.
- Round numbers normally count up; duplicates with different shas exist (`5867` has two `2-*`). Order duplicates by mtime. A deep re-review of an already-reviewed sha gets `<n+1>-<same-sha>`.
- `reviews/README.md` and root `index.html`: generated by `scripts/build-indexes.sh`, never hand-edited. The README is the fastest gno-PR index: status, title, author (open-PR tables), round count, reviewers, latest-round file links, aggregated AI verdict across all legacy formats. Check its `_Last updated:` line; reviews finished after that exist only on disk. Disk wins on disagreement.
- `reviews/security/<audit-*>/`: audit-harness output (`STATUS.md`, `raw-findings.jsonl`), explicitly unconfirmed. Label any hit from there as unverified, never as a review finding.
- `reviews/BATCH_STATUS.md`: transient batch-run state, possibly owned by a concurrent session. Treat as claim, verify on disk; during a batch, new rounds can land mid-query.

## File formats

Verdict, three formats: `**Verdict: X** — ...`; plain `Verdict: X — ...`; a `## Verdict` heading with the verdict on the next non-blank line (possibly bold, mixed case, e.g. `**Approve with minor suggestions.**`). Some lines carry a `> ` blockquote prefix or a dash instead of the colon. Vocabulary: APPROVE, REQUEST CHANGES, NEEDS DISCUSSION, CLOSE, plus legacy APPROVE WITH NITS and MERGED. A reanchored round may state the verdict only inside its `Round <n>.` note ("Verdict moves X -> Y"); when extraction yields nothing or nonsense, open the file.

Posted state, per round, from `comment*.md`: a `^Posted: <url>` line (written by the post script) means posted; a comment file without it is an unposted draft; no comment file means no draft. Absent `Posted:` means not posted, never guess from context. `[posted](...)` links mark which inline comments went out; `## SKIP` sections were deliberately withheld (their first body line `Already raised: <url>` names the duplicate they defer to).

The metadata header (`Author: <login> | Base: ... | Commit: <sha> (latest|stale — +N ...)`) is plain in old files, bold-wrapped (`**Author:**`) in new ones. Staleness was computed when scripts last ran; treat it as of the review date, not today.

## Modes

**PR lookup** (`$ARGUMENTS` is a number).

```bash
find reviews -maxdepth 3 -type d -name '<number>-*'
```

A bare number means gnolang/gno; when it also matches another repo's dir, report both and ask which. No hit from the find (it matches any slug): not reviewed. A hit whose directory holds no review file (`4817-*` holds only `overview.html`): report as present but reviewless, never as reviewed. Report, in order: PR dir; one line per round in mtime order (round dir, verdict, posted state); then the latest round's TL;DR and its headline findings by severity section. When rounds disagree on the verdict, the latest stands; name the overturned round.

**Search** (free text, including file paths and subsystems). Findings cite `file:line` anchors and comment drafts use `## <path>:<line>` headers, so a path fragment is a good term.

```bash
grep -rli '<term>' reviews/pr reviews/gnodaokit reviews/onboarding-bot --include='*.md'
```

Collapse hits to one line per PR, newest round first: PR number, latest verdict, the matching section's severity, a one-line quote, file link. Comment drafts duplicate review-file findings; count the PR once. Exclude repo `README.md` hits.

**author <login>**. PRs reviewed for a given author, from the metadata header:

```bash
grep -rlE --include='*.md' '^\*{0,2}Author:?\*{0,2} <login> \|' reviews/pr | cut -d/ -f1-4 | sort -u
```

Report one line per PR: number, title, latest verdict, posted state.

**stats**. Verdict distribution, posted/draft/no-draft counts, multi-round PRs. Compute from the listing loop (latest round per PR); counting every round double-counts re-reviews. Round counts and reviewer spread can be read off `reviews/README.md`.

**posted**. Every review actually sent, any round, with URL:

```bash
grep -r --include='comment*.md' -m1 -H '^Posted:' reviews/ | sed 's/:Posted: /\t/'
```

**unposted**. Findings raised but never sent, three sources, reported separately: latest-round drafts without `Posted:` (listing-loop rows marked `draft`); `^## SKIP` sections in comment drafts (withheld or already raised); `## Open questions` sections in review files (deliberately never drafted).

**recent [n]** (default 10). Latest rounds by mtime, each with verdict and posted state:

```bash
find reviews/pr reviews/gnodaokit reviews/onboarding-bot -mindepth 2 -maxdepth 3 \
  -type d -name '[0-9]*-*' -printf '%T@ %p\n' | sort -rn | head -<n> | cut -d' ' -f2-
```

## Listing loop

Canonical one-line-per-PR listing: `<pr> TAB <verdict> TAB <POSTED|draft|no-draft> TAB <latest-round-dir>`. Handles every naming generation, prefix-less files, no-round layouts, and `tests/` dirs; runs clean under zsh and bash. Scope non-gno repos by swapping the glob to `reviews/<repo>/*/`.

```bash
for d in reviews/pr/*/*/; do
  n=$(basename "$d"); n=${n%%-*}
  last=$(find "$d" -maxdepth 1 -type d -name '[0-9]*-*' | sort -V | tail -1)
  [ -z "$last" ] && last="${d%/}"
  rf=$(find "$last" -maxdepth 1 -name '*.md' ! -name 'comment*' ! -name 'overview*' \
       -exec grep -il 'verdict' {} + 2>/dev/null | sort | head -1)
  if [ -z "$rf" ]; then printf '%s\tNO-REVIEW-FILE\t-\t%s\n' "$n" "$(basename "$last")"; continue; fi
  v=$(awk '/^#+ *Verdict/{h=1;next} h&&NF{print;exit} /^(> )?\*{0,2}Verdict[:* —-]/{print;exit}' "$rf" |
      grep -oiE 'approve with nits|request changes|needs discussion|approve|merged|close' | head -1 |
      tr 'a-z' 'A-Z')
  c=$(find "$last" -maxdepth 1 -name 'comment*.md' | sort | head -1)
  if [ -z "$c" ]; then post=no-draft
  elif grep -q '^Posted:' "$c"; then post=POSTED
  else post=draft; fi
  printf '%s\t%s\t%s\t%s\n' "$n" "${v:-?}" "$post" "$(basename "$last")"
done
```

A `?` verdict means the file has a nonstandard verdict location (round note, prose); open it rather than reporting unknown.

## Rules

- Read-only. This skill never writes a review, edits a draft, posts, pushes, or regenerates indexes.
- Cite `reviews/...` paths as clickable links so the user can open the source.
- A finding's severity comes from the section it sits under (Critical, Warnings, Nits, Missing Tests, Suggestions). Report the section, never re-rank.
- Verdicts are per-round. Reviewer files inside one round can disagree too (parallel lenses); report the spread, never average.
- The corpus reflects the reviewed sha, not current master or the PR's current head. Before presenting a finding as live, check the head moved or say plainly that you have not checked.
- Never infer "unreviewed" from a missing directory under one glob; match on the leading number across all of `reviews/` first (slugs drift, repos differ).
- Prefer `find` over bare globs in ad-hoc commands; the interactive shell is zsh and an unmatched glob aborts the whole command line.
- Same PR number can exist in two repos (gno PR 65 vs `gnodaokit/65-*`); disambiguate before answering.
